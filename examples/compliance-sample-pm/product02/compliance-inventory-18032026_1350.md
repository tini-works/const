> **Constitution terminology**: `Confirmed` in this inventory = `Proven` in the Constitution. An item is either proven or it is not — binary, no intermediate states. Proof requires evidence (a test run, a review, a demonstration) recorded with a named human verifier and date.

**Detailed specifications**: [`compliances/`](../compliances/00-scope-and-overview.md) — phase-by-phase requirement breakdowns from all four compliance sources.

### Confirmation Criteria

| Status | Meaning | Required |
|--------|---------|----------|
| **Confirmed** | Proven end-to-end across all verticals (= Proven per the Constitution) | Ext. Source + Design Match + Engineer Status + QA Status + Confirmed By |
| **Suspect** | Was Confirmed but a drift or source change was detected — **not proven** until PM reviews (per the Constitution: no intermediate states) | Previously had all matches, now under review |
| **TBC** | To be confirmed — at least one downstream match missing. Not proven. | Partial matches exist |

### Verification Lifecycle

```
TBC ──(all matches filled)──→ Confirmed
 ↑                               │
 │                          (drift/change detected)
 │                               ↓
 └──(material change)─── Suspect ──(false alarm)──→ Confirmed
                               │
                          (source removed)
                               ↓
                             Stale ──(cleanup confirmed)──→ Deleted
```

- **TBC**: No proof yet. Default state for new items. (Constitution: unverified)
- **Confirmed**: All match columns filled + evidence recorded + `Confirmed By` with named human and date. Audit-ready. (Constitution: proven)
- **Suspect**: Was Confirmed, but something changed. **Not proven** until PM reviews. (Constitution: suspect — proof is expired)
- **Stale**: Source requirement removed. Kept temporarily for impact review, then deleted.
- **N/A**: Justified exclusion — must have a DEC entry explaining why.

### Requirement Type

| Value | Meaning |
|-------|---------|
| **Mandatory** | Required by the compliance framework — must be implemented in every contract |
| **Optional** | Conditionally required — applies only when activated by contract configuration, temporal rules, or feature availability |
| **Stale** | Was confirmed but upstream changed (new AKA catalog, requirement change) | Needs re-verification |
| **N/A** | Not applicable to PVS (handled by infrastructure, not offered as feature) | Justified exclusion |

## 3. Compliance Checklist — Master Inventory

This is the matchable checklist. Each row is one **box** that downstream verticals match against.

**Column schema:**
- **Obligation**: what must be true (the box) — written in PM language, not code
- **Requirement Type**: Mandatory / Optional
- **Ext. Source**: which regulation, catalog, or customer request requires this
- **Status**: TBC (not yet matched) / Confirmed (all matches verified) / Suspect (was Confirmed, drift detected — awaiting re-verification) / Stale (upstream source removed) / N/A (justified exclusion)
- **Verification Method**: what type of test proves this obligation holds — filled by PM when creating the row
- **User Story & AC**: combined column — As a {persona}, I want {obligation as desire}, so that {business justification}. **AC:** Given/When/Then acceptance criteria. Merges who needs it (persona), what they need (obligation), why (business goal), and how to verify (acceptance criteria) into one field per the Constitution user story pattern
- **Design Match**: screen or flow that satisfies this obligation. Simple format: `Screen > Sub-feature`. Expanded format for complex items: `Screen > Sub-feature — {interaction: loading/error/validation behavior}`. Use expanded format for multi-step workflows, screens with error handling, or items where the interaction pattern matters for compliance
- **Engineer Status**: implementation status for this obligation — filled by Engineering vertical
- **Engineer Ref**: reference to implementation (code path, service, module) — filled by Engineering vertical
- **QA Status**: test coverage status for this obligation — filled by QA vertical
- **QA Ref**: reference to test case that proves this obligation holds — filled by QA vertical
- **Reported Issues**: bug tickets, customer complaints, billing rejections, or audit findings linked to this item. Format: `{ticket-ID} — {root cause summary}`. For P0/P1 incidents, add post-mortem reference: `{ticket-ID} — {root cause} — PM:{post-mortem-ref}`. When a production incident traces to a compliance item, the root cause must be recorded here, not just the ticket number
- **Track**: progress shorthand for filtering — see Track Column Legend below

**Engineer Status values:**

| Value | Meaning |
|-------|---------|
| `—` | Not assessed |
| `Implemented` | Code exists and covers this obligation |
| `Partial` | Code exists but does not fully cover the obligation |
| `Not implemented` | No code exists for this obligation |
| `N/A` | Not applicable (handled by infrastructure or external system) |

**Engineer Ref format:** `service/package.FunctionName` or `service/package` (module-level). Examples: `service/billing.ValidateCode0000`, `app/app-core/api/enrollment.SubmitTE`

**QA Status values:**

| Value | Meaning |
|-------|---------|
| `—` | Not assessed |
| `Covered` | Test case exists and passes |
| `Partial` | Test exists but doesn't cover all acceptance criteria |
| `Failing` | Test exists but currently fails |
| `Not covered` | No test case exists |

**QA Ref format:** `TC-{SECTION}-{NNN} ({environment}, {date}, {result})` — e.g., `TC-ABRD-001 (staging, 2026-03-15, pass)`, `TC-KVDT-015 (localhost, 2026-04-01, fail)`. A TC-ID without execution metadata is a claim, not proof. Every QA Ref entry must include when it was last run, where, and whether it passed.

**Verification Method categories:**

| Method | When to use | Typical sections |
|--------|-------------|-----------------|
| **Validator unit test** | Rule-based checks on service codes, diagnoses, billing | ABRD, VSST, KBV EBM, GOÄ, KBV TSS, KBV ICD-10-GM, ICD-10-GM |
| **Integration test (API)** | End-to-end API flow that triggers validation | ABRG, VERT, KVDT Service, KVDT Billing, eDMP |
| **E2E workflow test** | Multi-step user journey across screens and services | VERE (enrollment lifecycle) |
| **Form output check** | Generated PDF/PCX matches template field-by-field | FORM, Forms/eAU/eArztbrief/ePA |
| **Data import validation** | Imported data (eGK, KVDT, contract XML) parsed correctly | PSDV, DETE, KVDT Patient Data |
| **Connector handshake test** | TI connector, KIM, gematik service communication | ITVE, E-Rezept |
| **Manual review** | Cannot be fully automated — requires human verification | ALLG (documentation, advertising, user manual) |

**User Story & AC guidance:**
- The "I want" derives from the Obligation column. The "AC:" derives by inverting the obligation into testable Given/When/Then conditions
- Example: Obligation "code 0000 must be prompted" → Criteria "Quarter with services but no code 0000 → error displayed; quarter with code 0000 → no error"
- Use `—` when Obligation is _(name pending)_ — fill criteria after obligation text is resolved

**Track Column Legend:**

| Value | Meaning |
|-------|---------|
| `_` | Not started — no matches filled |
| `D` | Design matched only |
| `E` | Engineer matched only |
| `Q` | QA matched only |
| `DE` | Design + Engineer matched |
| `DQ` | Design + QA matched |
| `EQ` | Engineer + QA matched |
| `DEQ` | All matched → Confirmed |

## 7. Reconciliation Log & Change Log

Per the Constitution, reconciliation is event-driven: each change event triggers an assessment before any modifications. This section contains both the reconciliation log (what was assessed) and the change log (what was modified).

Item-level audit trail for every compliance change. Each entry records what changed, when, why, and what needs re-verification.

### Change types

| Type | Meaning | Effect on existing rows |
|------|---------|------------------------|
| **ADDED** | New obligation from source update | New row created, Status = TBC, Track = `_` |
| **REMOVED** | Obligation no longer in source catalog | Row marked Status = `Stale`, then deleted after impact review |
| **CHANGED** | Obligation text or requirement type modified by source | Status reverts to `TBC`, Track reverts to `_`, all match columns reset to `—` |
| **RECLASSIFIED** | Requirement type changed (Mandatory ↔ Optional) but obligation unchanged | Status stays, match columns stay, Requirement Type updated |
| **SUPERSEDED** | Obligation replaced by a different ID in new catalog version | Old row marked Stale with pointer to new ID, new row created |

### Re-verification protocol

When a source update affects an item that was previously `Confirmed` (DEQ):

1. **CHANGED obligation** → Status set to `Suspect`. PM reviews the change. If the change materially affects the obligation, Status reverts to `TBC`, Track reverts to `_`, all match columns (Design, Engineer, QA) reset to `—`. If the change is cosmetic/non-material, Status returns to `Confirmed` with a DEC entry explaining why. The item must be re-verified from scratch only if the change is material.
2. **REMOVED obligation** → Status set to `Suspect`. PM reviews: if the source truly removed the obligation, Status moves to `Stale`. If the removal is a catalog reorganization (obligation moved, not deleted), PM traces to new location and updates the row. If the code/test should also be removed, link to a cleanup ticket in Reported Issues. Row is deleted after cleanup is confirmed.
3. **RECLASSIFIED** (Mandatory→Optional or vice versa) → Status set to `Suspect`. PM reviews whether the reclassification implies a scope change. If no scope change, Status returns to `Confirmed` with a DEC entry. If scope change is implied, Status reverts to `TBC` for re-verification. Only the Requirement Type column is updated in either case.
4. **SUPERSEDED** → Old row gets Status = `Suspect`. PM reviews: if the old obligation is truly replaced, Status moves to `Stale` with note `→ {NEW_ID}`. New row created with fresh TBC status. If the obligation is substantially the same, match columns can be copied from the old row as a starting point (but still need PM review). If the "supersession" is actually a renumbering with no content change, PM can mark old row Stale and new row Confirmed with a DEC entry.

### Cross-reference impact rules

When a source change in one section affects items in other sections:

| Source change | Check these sections for impact |
|--------------|-------------------------------|
| EBM code retired/changed | ABRD (billing doc), ABRG (billing process), KVDT Service (3.17), KVDT Billing (3.18) |
| ICD-10 code retired/changed | ABRD (diagnosis rules), ABRG (diagnosis in billing), ICD-10-GM (3.16), KVDT Service (3.17) |
| AKA service catalog changed | ABRD, ABRG, VERT, FORM, VSST — any section referencing contract-specific services |
| gematik spec changed | E-Rezept (3.19), Forms/eAU/ePA (3.20), ITVE (3.8) |
| KVDT field spec changed | KVDT Patient Data (3.15), KVDT Service (3.17), KVDT Billing (3.18) |

PM must check all impacted sections when processing a source change — not just the primary section.

### Impact Chain ID

When a single source change causes cascading effects across multiple sections, all related Change Log entries share the same **Impact Chain ID**. Format: `IC-{QUARTER}-{NNN}` (e.g., `IC-Q3-26-001`).

This allows tracing the full ripple of any change:
- "Show me everything caused by the EBM 03220 retirement" → filter by `IC-Q3-26-001`
- "How many chains were triggered in Q3?" → count distinct IC-Q3-26-* IDs
- "Which chain had the widest impact?" → count rows per chain ID

## 8. Drift Detection & Resolution

The Change Log (Section 7) tracks **planned changes** from compliance source updates. This section handles **unplanned discrepancies** — any drift between what's required (the obligation) and what's actually delivered (design, code, tests), regardless of where the drift originates.

### Drift types

| Type | What happened | Example |
|------|--------------|---------|
| **Compliance** | External requirement changed but inventory not yet updated | AKA publishes mid-quarter erratum; KBV issues urgent EBM correction |
| **Implementation** | Code refactored, moved, deleted, or restructured — Engineer Ref now points to dead path | `service/billing.Validate` renamed to `service/billing/v2.Validate`; service merged into another |
| **Specification** | Design/UX changed without updating inventory — Design Match no longer reflects actual screens | Billing screen redesigned; workflow reordered; field removed from UI |

### Severity

| Level | Meaning | Action required |
|-------|---------|----------------|
| **Critical** | Blocks certification — obligation is not met by current delivery | Must resolve before audit submission |
| **Major** | Breaks functionality — obligation is partially met or test is failing | Must resolve before release |
| **Minor** | Cosmetic or non-blocking — reference is stale but functionality still works | Resolve in next sprint |

### Re-verification trigger rules

When a drift is logged, affected inventory items are automatically flagged:

| Drift type | Status reverts to | Track reverts to | Columns reset | Reason |
|-----------|-------------------|-----------------|---------------|--------|
| **Compliance** | `Suspect` → PM reviews → `TBC` if confirmed | `_` | All match columns → `—` | Requirement itself changed, everything must be re-verified. Goes to Suspect first so PM can assess severity before full reset |
| **Implementation** | `Suspect` → PM reviews → `TBC` if confirmed | `D` (Design still valid) | Engineer Status, Engineer Ref, QA Status, QA Ref → `—` | Design still matches the obligation, but code and tests need re-verification. PM reviews whether the implementation change actually affects compliance |
| **Specification** | `Suspect` → PM reviews → `TBC` if confirmed | `_` | All match columns → `—` | If the design changed, the implementation and tests may no longer align either. PM reviews before full reset |

### Detection methods

| Method | Who runs it | When | Catches |
|--------|------------|------|---------|
| **Quarterly reconciliation** | PM | Every quarter (Section 6) | Compliance drift |
| **Code review / PR review** | Engineering | Every PR that touches files referenced in Engineer Ref | Implementation drift — reviewer checks if changed files are in the inventory |
| **QA test failure** | QA | Continuous — when a previously passing test starts failing | Implementation drift — code changed but test wasn't updated, or vice versa |
| **Design review** | PM | When UI/UX changes are merged | Specification drift — screens changed but inventory not updated |
| **Periodic spot-check** | PM | Quarterly — 10% sample of Confirmed items | All three drift types — manual verification that DEQ items are still valid |
| **CI pipeline hook** _(future)_ | Automated | On every commit | Implementation drift — detect when files listed in Engineer Ref column are modified |

### Drift resolution cycle

```
DETECT          Someone notices a discrepancy (any detection method above)
    │
    ▼
LOG             Create entry in Drift Register (below) with affected IDs, type, severity
    │
    ▼
FLAG            Revert affected items per Re-verification trigger rules (above)
                Update Track column, update per-section progress summary
    │
    ▼
NOTIFY          PM reviews the drift — is this a one-off or a systemic issue?
                If systemic (e.g., full service refactor), escalate to bulk re-verification
    │
    ▼
RE-VERIFY       Walk through the match columns again: Design → Engineer → QA
                Same process as Phase 2 (Progress) in the instructions
    │
    ▼
CLOSE           Drift Register entry → Status = Resolved
                Inventory items back to Confirmed (DEQ) if all matches pass
                Update Section 1 dashboard
```

|----------|------|------------|----------------|------|----------|-------------|------------|--------|

> **How to add entries**: When any discrepancy is found between the inventory and actual delivery, add a row immediately. Do NOT wait for quarterly reconciliation — drifts should be logged as soon as detected. The Drift ID format is `DR-{NNN}` (sequential). After logging, follow the resolution cycle above.

### Linking Drift Register to Change Log

- A **Compliance drift** that leads to inventory changes should ALSO create a Change Log entry in Section 7 (the drift triggers the change)
- An **Implementation drift** or **Specification drift** does NOT create a Change Log entry — the compliance source didn't change, only the delivery did
