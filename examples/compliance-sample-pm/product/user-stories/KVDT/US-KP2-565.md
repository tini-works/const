### User Story

As a **Abrechnungsverantwortliche**, I want **to capture and transfer the Behandlungstag bei IVD-Leistungen (FK 4214) for Muster 10, 10A, and 39 referrals**,
so that **the day of Probenentnahme is correctly used as Behandlungstag for billing rule application per EBM Allgemeine Bestimmungen 3.8.5**.

### Compliance Context

Per EBM 3.8.5, for in-vitro-diagnostic services the Probenentnahme day counts as Behandlungstag, independent of when the analysis was performed. This distinction is critical for applying billing rules like "once per Behandlungstag". Without FK 4214, the wrong date reference may be used.

### Acceptance Criteria

**Scenario 1: Behandlungstag capture prompted for matching referrals**
Given a referral matching Muster 10 (Satzart 0102 / Scheinuntergruppe 27), Muster 10A (Scheinuntergruppe 28), or Muster 39a (Scheinuntergruppe 21) is captured
When the user creates the billing record
Then the software requests the user to adopt the Behandlungstag of the referrer

**Scenario 2: Behandlungstag transferred in FK 4214**
Given the user has captured the Behandlungstag
When the billing is created
Then the software transfers the Behandlungstag in field FK 4214

**Scenario 3: Priority-based automatic capture**
Given the software supports automatic capture of Behandlungstag
When determining the value for FK 4214
Then the following priority applies:
  - 1st priority: Probenentnahmedatum
  - 2nd priority (if Probenentnahmedatum unknown): Ausstellungsdatum or Probeneingangsdatum

**Scenario 4: Automatic capture from Probenentnahmedatum**
Given a Probenentnahmedatum for the referral is known
When automatic capture is triggered
Then the software may automatically capture and transfer the Probenentnahmedatum in FK 4214

**Scenario 5: User can adjust automatically adopted date**
Given a date has been automatically adopted for FK 4214
When the user reviews the value
Then the user has the ability to adjust the automatically adopted date

**Scenario 6: Manual entry shows priorities**
Given the user is manually entering the date for FK 4214
When the entry field is active
Then the software shows the user the priorities per acceptance criterion 3

**Scenario 7: Warning when FK 4214 exceeds FK 5000**
Given the user has entered a date for FK 4214
When the date of FK 4214 is greater than the date of FK 5000
Then the software displays a warning
And the date of FK 4214 must not lie in the future

**Scenario 8: Warning when FK 4214 more than 60 days before FK 5000**
Given the user has entered a date for FK 4214
When the date of FK 4214 is more than 60 days less than the date of FK 5000
Then the software displays a warning that the difference should not be greater than 60 days
