### User Story

As the **System**, the system must **verify the cost carrier's obligation to provide services by checking insurance coverage dates when reading an insurance card, rejecting invalid cards with an error message while preserving data viewability**,
so that **only valid insurance cards are accepted for billing and form issuance, preventing claims against non-obligated cost carriers**.

### Compliance Context

Anlage 4a BMV-Ae requires verification of insurance coverage when reading cards. Accepting an expired or not-yet-valid card for billing would result in claims against a cost carrier with no obligation, leading to billing rejections and potential financial loss for the practice.

### Acceptance Criteria

**Scenario 1: Valid eGK -- coverage dates in range**
Given an eGK is read
And the read-in date (FK 4109) >= VersicherungsschutzBeginn
And the read-in date <= VersicherungsschutzEnde (if element present)
When the system validates the card
Then the card is valid
And reading the data must be possible
And billing and issuance of panel physician forms are permissible

**Scenario 2: Invalid eGK -- coverage dates out of range**
Given an eGK is read
And the read-in date (FK 4109) < VersicherungsschutzBeginn or > VersicherungsschutzEnde
When the system validates the card
Then reading the data must be rejected with an appropriate error message
And direct further processing for ADT billing and form issuance is not permissible
And the user must retain the ability to view rejected card data in copyable format per KP2-102
And no Einlesedatum per P2-140 must be generated

**Scenario 3: Valid KVK -- within expiry date**
Given a KVK is read
And the GueltigkeitsDatum (Objekttag '8D') is transformed to JJJJMMTT format
And the read-in date (FK 4109) <= VersicherungsschutzEnde
When the system validates the card
Then the card is valid and reading must be possible

**Scenario 4: Invalid KVK -- expired**
Given a KVK is read
And the read-in date (FK 4109) > VersicherungsschutzEnde
When the system validates the card
Then reading the data must be rejected with an appropriate error message
And direct further processing for billing and form issuance is not permissible
