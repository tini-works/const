### User Story

As a **System**, the system must be able to perform real-time system-side changes to GNRs during service documentation when check conditions of the EBM master data file are violated, while ensuring the user can recognize and reverse these changes,
so that billing data quality is improved at the point of entry without overriding the physician's clinical judgment.

### Compliance Context

Real-time checks against EBM conditions during service recording help physicians avoid incorrect entries. However, the physician must always be able to see and reverse system changes. Importantly, rule mechanisms (section `//gnr/regel/`) may only trigger advisory messages, not system-side changes -- only condition violations (section `//gnr/bedingung/`) may trigger automatic corrections.

### Acceptance Criteria

**Scenario 1: System-side changes on condition violations in real time**
Given the user is documenting services
When a check condition (`//gnr/bedingung/` of the EBM master data file) is violated
Then the software may perform system-side changes to the documented GNRs in real time

**Scenario 2: User can recognize and reverse changes**
Given the system has made changes to the service documentation
When the changes are applied at the time of recording
Then the user can recognize the system-side changes at the time they occur
AND the user can reverse the changes
