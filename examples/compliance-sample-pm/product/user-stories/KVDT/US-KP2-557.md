### User Story

As a **System**, The system must **automatically adopt changes to official insured person data (name, gender, date of birth, address) into all existing 010x billing records of the current quarter when the insurance card is re-read**,
so that **billing data remains consistent with the latest official insured person data across all active cases**.

### Compliance Context

Changes to registration data of insured persons lead to updated insurance cards or Online-VSDM updates. Billing records must reflect the latest official data to avoid rejections by the KV. Failure to propagate changes across all active records in the quarter creates data inconsistencies.

### Acceptance Criteria

**Scenario 1: Official fields adopted unchanged**
Given an insured person's insurance card is read
When official insured person data fields per Table 4 are present
Then the software adopts the official fields unchanged

**Scenario 2: Automatic adoption of changed data across all 010x records**
Given in the current quarter for the same patient the Besondere Personengruppe, insurer, and Versichertenart remain unchanged
When the insurance card is read and changes to fields 3100, 3101, 3102, 3103, 3104, 3107, 3109, 3110, 3112, 3113, 3114, 3115, 3120, 3121, 3122, 3123, and 3124 are detected due to name, gender, date of birth, or address change
Then the software automatically adopts these changes into all existing 010x data records of the patient in the current quarter

**Scenario 3: No new 010x record created on automatic adoption**
Given automatic data adoption per Scenario 2 occurs
When the changes are applied
Then the software does not automatically create a new 010x data record

**Scenario 4: Einlesedatum updated on automatic adoption**
Given automatic data adoption per Scenario 2 occurs
When the changes are applied
Then the software updates the Einlesedatum in all 010x data records belonging to the insured person in the current quarter
