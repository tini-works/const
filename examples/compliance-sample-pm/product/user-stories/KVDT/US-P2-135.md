### User Story

As the **System**, the system must **correctly transform and adopt the WOP identifier from KVK and eGK cards into field 3116, and allow manual entry without pre-population**,
so that **the residence identifier is accurately transmitted for billing per Anlage 4a BMV-Ae**.

### Compliance Context

The WOP (Wohnort-Kennzeichen) must not be determined programmatically to avoid incorrect assignments. Anlage 4a BMV-Ae and its Technische Anlage define the transformation specifications. Incorrect WOP assignment can lead to billing routed to the wrong KV region.

### Acceptance Criteria

**Scenario 1: WOP transformation from KVK**
Given a KVK is read
When the software processes the VKNR/WOP field (Objekttag '8F')
Then the content must be transformed per the "Technische Anlage zur Anlage 4a" [KBV_ITA_VGEX_Mapping_KVK] and transmitted in field 3116
And if no WOP identifier is coded on the KVK, field 3116 is not transmitted

**Scenario 2: WOP adoption from eGK**
Given an eGK is read
When the software processes the WOP element
Then the content of `../WOP` must be adopted per the "Technische Anlage zur Anlage 4a" into field 3116

**Scenario 3: Manual entry without pre-population**
Given the user enters data manually
When the WOP identifier input is displayed
Then the software must provide the ability to manually enter the WOP identifier
And the input field must not be pre-populated (e.g., based on the PLZ)

**Scenario 4: Transmission of manually entered WOP**
Given the user has manually entered a WOP identifier
When billing data is generated
Then the software must transmit the entered WOP identifier in field 3116
