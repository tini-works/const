### User Story

As a **System**, the system must perform final checks on GNRs based on EBM conditions and rules at the end of reference periods, while allowing the user to recognize and reverse changes and permitting storage against EBM conditions when needed,
so that billing data is validated comprehensively before submission while preserving physician override authority.

### Compliance Context

Final checks at the end of reference periods (treatment day, cycle case, etc.) provide a comprehensive validation of billing data. Unlike real-time checks, final checks may use both conditions and rules from the EBM master data file. Critically, the system must still allow storage against EBM conditions and rules, preserving the physician's authority to bill as they deem appropriate.

### Acceptance Criteria

**Scenario 1: System-side changes based on conditions and rules**
Given a reference period (treatment day, cycle case) has ended
When the final check is performed
Then the system may make system-side changes to GNRs based on check conditions and rules of the EBM master data file

**Scenario 2: User can recognize and reverse changes**
Given the system has made changes during the final check
When the changes are applied
Then the user can recognize the system-side changes to service documentation
AND the user can reverse the changes if applicable

**Scenario 3: Allow storage against EBM conditions and rules**
Given the user wants to store services that violate EBM conditions or rules
When the billing file is generated
Then the storage of services in the ADT billing file is possible against the check conditions and rules of the EBM master data file
