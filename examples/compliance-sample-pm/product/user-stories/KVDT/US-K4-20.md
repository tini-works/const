### User Story

As a **System**, the system must perform plausibility checks on the Identifier-SA for pregnancy termination billing and warn the user of implausible entries while still allowing storage after user confirmation,
so that billing data quality is maintained without blocking legitimate but unusual entries.

### Compliance Context

The Identifier-SA is a multi-part identifier used in pregnancy termination billing that encodes patient origin, application date, running number, IK, PLZ, and internal identifier. Plausibility checks on each component help catch data entry errors early. However, the system must not block storage of implausible identifiers if the user explicitly confirms, since edge cases may exist.

### Acceptance Criteria

**Scenario 1: Warn on implausible digit 1 (patient origin)** [derived]
Given the user enters an Identifier-SA with digit 1 = 0
When the plausibility check is performed
Then a warning is issued that the case is to be billed directly with the responsible health insurance fund

**Scenario 2: Validate date format in digits 2-7** [derived]
Given the user enters an Identifier-SA
When the plausibility check is performed on digits 2-7
Then the system checks for valid DDMMYY date format
AND verifies the application date is <= 1st service day

**Scenario 3: Validate IK and PLZ via master files** [derived]
Given the user enters an Identifier-SA
When the plausibility check is performed
Then digits 9-15 (IK) are checked via KT master file
AND digits 16-20 (PLZ) are checked via PLZ master file

**Scenario 4: Allow storage of implausible identifier after confirmation**
Given the plausibility check has found implausible entries
When the user confirms the entry
Then the implausible identifier is stored
