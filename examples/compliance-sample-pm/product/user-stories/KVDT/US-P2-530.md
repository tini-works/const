### User Story

As a **System**, the system must automatically create a new "010x" data record when a patient's VKNR/KTAB combination changes in the current quarter,
so that billing to multiple cost units for the same patient is correctly separated.

### Compliance Context

The combination of VKNR (FK 4104) and KTAB (FK 4106) uniquely identifies a cost unit. When a patient switches insurance mid-quarter, services must be billed to different cost units. Without automatic record separation, services could be billed to the wrong insurer.

### Acceptance Criteria

**Scenario 1: New 010x record on VKNR-KTAB change**
Given a patient's VKNR (FK 4104) - KTAB (FK 4106) combination changes in the current quarter
When the change is detected
Then the software generates an additional "010x" data record

**Scenario 2: No new record for Kostenträgerkennung-only change**
Given only the Kostenträgerkennung (FK 4111) changes but the Abrechnungs-VKNR does not change
When the change is detected
Then the software does not create a new "010x" data record

**Scenario 3: Preserve existing Einlesedatum on insurance switch**
Given a new 010x data record is created due to an insurance switch
When the new record is generated
Then no changes to the already existing Einlesedatum (FK 4109) of previous records take place
AND the Einlesedatum is updated only for the newest data record
AND the Einlesedatum is stored and transmitted "true to data record" (see P2-150)
