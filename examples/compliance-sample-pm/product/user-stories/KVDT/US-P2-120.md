### User Story

As the **System**, the system must **treat insurance card data as "official" (immutable) for the quarter in which the card was read, storing and transmitting it unchanged with the billing**,
so that **billing data matches the insurance card exactly and is not rejected by health insurers**.

### Compliance Context

Modification of read insurance card data from a valid card can cause rejections in the billing process by health insurers. The "official" character of card data persists until end of the quarter. This requirement stems from Anlage 4a BMV-Ae and its Technische Anlage.

### Acceptance Criteria

**Scenario 1: Official data fields after successful card reading**
Given an insurance card has been successfully read per the "Technische Anlage zur Anlage 4a"
And a read-in date has been generated per P2-140
When the data is stored
Then all listed personal insurance data, protected insurance data, street address, PO box address, and general insurance data fields have "official" character
And requirements P2-105, P2-135, P2-136, P2-170, KP2-310, P2-470 must be observed

**Scenario 2: Immutability of official data**
Given fields with official character exist for a patient
When the user or a programmatic process attempts to modify these fields
Then the software must prevent the modification through both UI and programmatic means

**Scenario 3: Official data validity until end of quarter**
Given official insurance card data exists
When the quarter in which the card was read ends
Then the officiality of the data expires

**Scenario 4: Prevention of automatic adoption into substitute procedure**
Given a successful reading process of an insurance card has occurred
When the system processes the data
Then the software must prevent the automatic adoption of insurance card data into the substitute procedure for the purpose of data modification
