### User Story

As the **System**, the system must **automatically generate a read-in date (Einlesedatum) from the system date upon successful insurance card reading via a stationary reader, display it as read-only, and prevent manual generation or modification**,
so that **the read-in date accurately proves when the card was read, as required for billing legitimation**.

### Compliance Context

The read-in date (FK 4109) serves as proof that an insurance card was presented and read during treatment per section 13 BMV-Ae. If the date can be manipulated or manually created, the integrity of the billing legitimation chain is compromised, potentially enabling fraud.

### Acceptance Criteria

**Scenario 1: Automatic generation of read-in date from stationary reader**
Given a successful reading process of an insurance card occurs through a stationary card reader
When the card data is processed
Then a system date is automatically provided by the operating system
And this date is displayed on screen as the "Einlesedatum" (FK 4109)

**Scenario 2: Read-in date is read-only**
Given a read-in date has been generated
When the user views the field
Then the read-in date has "official" character and is a read-only field
And the user must not be able to modify this field

**Scenario 3: No manual generation of read-in date**
Given insurance card data is entered manually via keyboard, scanner, or other input medium
When the data is processed
Then there must be no possibility to generate or manually enter the read-in date

**Scenario 4: Read-in date from mobile card terminal**
Given insurance card data is adopted from a mobile card terminal (return code 9500 or 9501 at RESET_CT)
When the data is processed
Then the Einlesedatum is adopted from the data object Einlesedatum (tag '91')
And the Zulassungsnummer is adopted from the data object Zulassungsnummer (tag '92')

**Scenario 5: Imported FK 4109 is immutable**
Given record type 8215 "Auftrag" (LDT 3) imports field 4109 into the patient documentation
When the data is displayed
Then the user must not be able to modify the content of field 4109
