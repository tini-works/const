## P3-640 — Multiple Prescription

| **ID** | P3-640 |

### Requirement

The prescribing software must enable the creation of a multiple prescription (§ 31 Absatz 1b SGB V in conjunction with § 4 Absatz 3 AMVV) within the scope of an electronic pharmaceutical prescription (pursuant to [P3-710](P3-710.md)).

### Rationale

The multiple prescription provides that physicians may issue prescriptions that permit up to three repeat dispensings after the initial dispensing. The prescriptions are to be specifically marked. The aim is to ensure longer-term supply for insured persons who require continuous treatment with a specific pharmaceutical. An appropriate minimum interval between the redemption deadlines of the respective partial prescriptions, commensurate with the package size, should be provided.

### Acceptance Criteria

1. The prescribing software must provide the contracted physician with the option to create a multiple prescription exclusively for an electronic pharmaceutical prescription (pursuant to [P3-710](P3-710.md)).
   - The content of the individual FHIR instances of the partial prescriptions is identical. Only the contents of MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription) differ in accordance with acceptance criteria 3, 4, and 5.
2. The prescribing software must automatically set the "Kennzeichen Mehrfachverordnung" (multiple prescription marker) for the multiple prescription.
   - The marker indicating whether the current prescription is part of a multiple prescription is set via the element MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription).Kennzeichen.valueBoolean in profile KBV_PR_ERP_Prescription.
   - The system must set the "Kennzeichen Mehrfachverordnung" to `true` if the current prescription is part of a multiple prescription.
   - The system must set the "Kennzeichen Mehrfachverordnung" to `false` if no multiple prescription is present.
3. The number of partial prescriptions within a multiple prescription is to be specified by the physician. A selection of two to four partial prescriptions must be enabled.
   - One data record for the electronic pharmaceutical prescription is to be generated per partial prescription of the multiple prescription.
   - The data records are to be automatically numbered sequentially by the software in the numerator of the numbering.
     - The values 1, 2, 3, or 4 are permitted.
     - The numerator must not be higher than the denominator.
     - The numerator is specified via the element MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription).Nummerierung.valueRatio.numerator.value in profile KBV_PR_ERP_Prescription.
   - The total number of partial prescriptions provides the denominator of the numbering.
     - The values 2, 3, or 4 are permitted.
     - The denominator is specified via the element MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription).Nummerierung.valueRatio.denominator.value in profile KBV_PR_ERP_Prescription.
4. The software must enable the contracted physician to specify the start of the redemption period for each partial prescription.
   a. The start of the redemption period must not be before the issue date of the partial prescription.
   b. The respective starts of the redemption period for the individual partial prescriptions must not be identical.
   c. The software must ensure that the start of the redemption period specified by the contracted physician is not more than one year (365 days) after the issue date.
   d. The start of the redemption period is specified via the element MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription).Zeitraum.valuePeriod.start in profile KBV_PR_ERP_Prescription.
   e. Specifying the start of the redemption period is mandatory for the contracted physician.
   f. The software must enable the contracted physician to automatically pre-populate the start of the redemption period per partial prescription. The contracted physician must be able to configure this pre-population.
      i. The software must enable the contracted physician to at least configure, activate, or deactivate this function for the entire system.
      ii. The configuration of the start of the redemption period must be possible independently of the configuration of the end of the redemption period.
      iii. An initial pre-population of the configuration may be made by the software.
      iv. Manual modification of the start of the redemption period by the contracted physician must be possible at any time.
      v. The requirements stated in this acceptance criterion under points a through e also apply to pre-population through configuration.
5. The software must enable the contracted physician to specify the end of the redemption period for each partial prescription.
   a. The end of the redemption period must be within a partial prescription (a data record) equal to or after the date of the start.
   b. The software must ensure that the end of the redemption period specified by the contracted physician is not more than one year (365 days) after the issue date.
   c. The end of the redemption period is specified via the element MedicationRequest.extension(https://fhir.kbv.de/StructureDefinition/KBV_EX_ERP_Multiple_Prescription).Zeitraum.valuePeriod.end in profile KBV_PR_ERP_Prescription.
   d. Specifying the end of the redemption period is not mandatory for the contracted physician.
   e. The software must enable the contracted physician to automatically pre-populate the end of the redemption period per partial prescription. The contracted physician must be able to configure this pre-population.
      i. The software must enable the contracted physician to at least configure, activate, or deactivate this function for the entire system.
      ii. The configuration of the end of the redemption period must be possible independently of the configuration of the start of the redemption period.
      iii. An initial pre-population of the configuration may be made by the software.
      iv. Manual modification of the end of the redemption period by the contracted physician must be possible at any time.
      v. The requirements stated in this acceptance criterion under points a through d also apply to pre-population through configuration.
6. The software must enable the specification of the start and end of the redemption period by the contracted physician independently of each other.
