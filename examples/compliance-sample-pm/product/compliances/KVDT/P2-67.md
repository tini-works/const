## P2-67 — Transmission of Proof of Support for TI Applications and TI Component Equipment

| Field | Value |
|-------|-------|
| **ID** | P2-67 |
| **Type** | Mandatory |
| **Source** | KVDT v6.06 |
| **Section** | 2.1.5 Benutzer- und Betriebsstättenverwaltung |
| **Status** | TBC |
| **Goals** | — |
| **Verification Method** | — |
| **Matched by** | — |
| **Refer** | [P2-66](P2-66.md) |

### Requirement

The software must enable the transmission of proof of support for functional features of TI applications or TI component equipment by a primary system available at the practice site into the ADT billing.

### Rationale

So that the responsible Kassenärztliche Vereinigungen (KVen) can ensure the provisions for payment of TI lump sums from the "Festlegung des Vereinbarungsinhalts durch das Bundesministerium für Gesundheit gemäß § 378 Absatz 2 Satz 2 in Verbindung mit Absatz 3 und 4 des Fünften Buches Sozialgesetzbuch (SGB V)", it is necessary that the required information be transmitted from the practices within the scope of billing.

### Acceptance Criteria

1. The software must, as a general rule for ADT billing, automatically pre-populate the TI application/TI component (FK 0225) with the values "eRezept" (1), "NFDM" (3), "eMP" (4), "KIM" (5), "eAU" (6), "eArztbrief" (7), "Kartenterminal" (8), "SMC-B" (9), "eHBA" (10), "eVDGA" (12), and TIM (13) and allow the user to review these before adoption into the ADT billing.
2. The software must check whether information about a primary system available at the practice site supports the functional features of the TI applications eRezept, NFDM, eMP, KIM, eAU, eArztbrief, eVDGA, and TIM, and whether the practice site has at least one card terminal, at least one SMC-B, and at least one eHBA available, in each case in Field FK 0226 (system support/practice equipment) automatically pre-populate and allow the user to review these before adoption into the ADT billing.
   a) The currently available information within the billing quarter must be pre-populated in daily steps.
   b) If the software does not support the respective TI application or the practice site is not equipped with the respective TI component (FK 0226 = 0) and in the previous quarter for the same practice site number or hospital IK the value "ja" (1) was captured in the ADT billing, the value "ja" (1) must be pre-populated.
   c) If the software does not support the respective TI application or the practice site is not equipped with the respective TI component (FK 0226 = 0) and in the previous quarter for the same practice site number or hospital IK the value "nein" (0) or no value was captured in the ADT billing, the value "nein" (0) must be pre-populated.
   d) When a user has not actively acknowledged the expression (combination FK 0225 and 0226) of the functional features of the TI application/TI component and these expressions have changed compared to the transmitted values of the previous quarter's billing, the software notifies the user of these changes before the transmission of the ADT billing data record.
3. The software does not allow the user to modify the values pre-populated according to Acceptance Criterion 1 for the TI application/TI component (FK 0225) prior to storing the data in the practice site data record ("besa") per practice site number or hospital IK in the KVDT file.
4. The software must allow the user to modify the values pre-populated according to Acceptance Criterion 2 for system support/practice equipment (FK 0226) prior to storing the data in the practice site data record ("besa") per practice site number or hospital IK in the KVDT file.
5. The software must check whether a primary system available at the practice site supports the functional features of the TI application ePA, supports specific capture, and allows the user to enable this before adoption into the ADT billing for acknowledgment and **adjustment**.
   a) Possible values for support are "keine ePA Unterstützung" (no ePA support) and "Unterstützung von ePA Stufe 3 (ePA4all)" (support of ePA Stage 3).
   b) The currently available information within the billing quarter must be pre-populated in daily steps.
   c) If the software does not support the TI application ePA, then the software adopts the user's selection from the previous quarter.
      i. If no value from the previous quarter is known, the software pre-populates "keine ePA Unterstützung" (no ePA support).
   d) When a user has not actively acknowledged the functional features of the TI application ePA and the values have changed compared to the transmitted values of the previous quarter's billing, the software notifies the user of these changes before the transmission of the ADT billing data record.
6. The software transmits per practice site number or hospital IK the values determined from Acceptance Criterion 5 into the billing as follows:
   a) Either "keine ePA Unterstützung" (no ePA support):
      i. FK 0225 = 11
      ii. FK 0226 = 0
   b) Or "Unterstützung von ePA Stufe 3 (ePA4all)" (support of ePA Stage 3):
      i. FK 0225 = 11
      ii. FK 0226 = 1
7. Since the communication between PVS and connector is the prerequisite for the transmission of the proof, the successful completion of the gematik confirmation procedure must be demonstrated and the confirmation letter -- Bestätigung der Konformität des Primärsystems zur Konnektorschnittstelle: Funktionsumfang VSDM -- must be submitted to the KBV as part of the "1ClickAbrechnung" certification.
