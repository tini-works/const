### Requirement

1. Following a successful reading process of an insurance card through a **stationary** card reader, a system date is automatically provided by the operating system.
2. The system date may only be generated, displayed, and further processed upon a successful reading process of an insurance card.
3. This system date is displayed on screen as the "Einlesedatum" (read-in date) of the insurance card (= "letzter Einlesetag der Versichertenkarte im Quartal" (last read-in day of insurance card in quarter), FK 4109).
4. This read-in date has "official" character and is a read-only field. The user must not be able to modify this field.
5. The read-in date may only be adopted into the patient master data and stored under the above conditions.
6. For billing, the read-in date is stored in the field "Letzter Einlesetag der Versichertenkarte im Quartal" (FK 4109 of the (x)ADT data package).
7. There must be **no possibility** to generate or manually enter the read-in date when insurance card data is entered manually via keyboard (e.g. in the substitute procedure), via scanner, or via any other input medium.
8. When record type 8215 "Auftrag" (LDT 3) imports field 4109 "letzter Einlesetag der Versichertenkarte im Quartal" into the patient documentation of a billing system, the user must not be able to modify the content of field 4109.
9. After adopting the insurance card data record from a **mobile** card terminal (return code 9500 or 9501 at RESET_CT), the **Einlesedatum** is not provided by the system date but adopted from the data object Einlesedatum (tag '91'), as is the **Zulassungsnummer** (approval number) of the mobile card terminal from the data object Zulassungsnummer (tag '92'), cf. Chapter "Mobiles Einsatzszenario" in "Mappingtabelle_KVK" [KBV_ITA_VGEX_Mapping_KVK] and also [P2-180](P2-180.md).
