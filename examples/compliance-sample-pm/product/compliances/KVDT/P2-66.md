### Requirement

The software must enable the transmission of the "compatibility-relevant components of the connector product type version" into the ADT billing.

### Rationale

The availability and on-demand use of a connector in a panel practice corresponding to a specific product type version is a necessary prerequisite for the reimbursement of lump sums pursuant to Anlage 32 BMV-Ä by the responsible Kassenärztliche Vereinigung, which must verify these. Furthermore, pursuant to § 341 Absatz 6 SGB V, service providers participating in panel care must demonstrate to their respective responsible Kassenärztliche Vereinigungen that they have access to the required components and services for the electronic patient record. By transmitting the "product type version of the connector" into the ADT billing, the Kassenärztliche Vereinigungen are provided with a simple means of verifying this prerequisite and obtaining this proof.

### Acceptance Criteria

1. The software must transfer the compatibility-relevant components of the product type version of connectors operated in a (subsidiary) practice site into the billing and store them in Field FK 0224 in the practice site data record ("besa") per practice site number or hospital IK in the KVDT file, if this information is retrievable via the external interface.
   a) The currently available value within the billing quarter must be adopted in daily steps.
   b) If more than one connector is operated in a (subsidiary) practice site, the compatibility-relevant components of the product type version with the numerically highest value must be adopted.
2. The software does not allow the user to manually enter the compatibility-relevant components of the product type version.
3. The software does not allow the user to modify the values pre-populated according to Acceptance Criterion 1 for the TI application/TI component (FK 0225) prior to storing the data in the practice site data record ("besa") per practice site number or hospital IK in the KVDT file.
4. The software must allow the user to modify the values pre-populated according to Acceptance Criterion 2 for system support/practice equipment (FK 0226) prior to storing the data in the practice site data record ("besa") per practice site number or hospital IK in the KVDT file.
5. The software must check whether a primary system available at the practice site supports the functional features of the TI application ePA, supports specific capture, and allows the user to enable this before adoption into the ADT billing for acknowledgment and **adjustment**.
   a) Possible values for support are "keine ePA Unterstützung" (no ePA support) and "Unterstützung von ePA Stufe 3 (ePA4all)" (support of ePA Stage 3).
   b) The currently available information within the billing quarter must be pre-populated in daily steps.
   c) If the software does not support the TI application ePA, then the software adopts the user's selection from the previous quarter.
      i. If no value from the previous quarter is known, the software pre-populates "keine ePA Unterstützung" (no ePA support).
   d) When a user has not actively acknowledged the functional features of the TI application ePA and the values have changed compared to the transmitted values of the previous quarter's billing, the software notifies the user of these changes before the transmission of the ADT billing data record.
6. The software transmits per practice site number or hospital IK the values determined from Acceptance Criterion 5 into the billing as follows:
   a) Either "keine ePA Unterstützung" (no ePA support):
      i. FK 0225 = 11
      ii. FK 0226 = 0
   b) Or "Unterstützung von ePA Stufe 3 (ePA4all)" (support of ePA Stage 3):
      i. FK 0225 = 11
      ii. FK 0226 = 1

### Notes

The pre-population or adjustment of field codes can be done individually per practice site number or hospital IK, or for a group of practice site numbers or hospital IK, collected e.g. by the software based on technical and/or by the user based on administrative criteria.

Since Field 0226 according to the KVDT data record description represents a hierarchy level deeper directly below Field 0225, the conditional mandatory fields 0225 and 0226 always form a pair. According to Acceptance Criteria 1, 2, 5, and 6, twelve pairs per practice site must be transmitted in the billing, where Field 0225 must contain the values "eRezept" (1), "NFDM" (3), "eMP" (4), "KIM" (5), "eAU" (6), "eArztbrief" (7), "Kartenterminal" (8), "SMC-B" (9), "eHBA" (10), "ePA Stufe 3" (11), "eVDGA" (12), and TIM (13).
