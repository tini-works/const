## Overview

The Secrets Engine manages runtime secrets injection and rotation via HashiCorp Vault. Secrets are injected into pods via the `vault-agent-injector` sidecar — never baked into images. Rotation happens in Vault with a rolling restart of sidecars, not a full redeploy. The engine tracks which services have picked up new secret versions and alerts when any service is still running on an expired version.

## Vault Path Convention

Each service gets a Vault path per environment:

```
secret/data/pave/{service_name}/{environment}/
    redis-password
    db-url
    stripe-key
    ...
```

**Vault policies** are generated by the Secrets Engine when a service is registered:

```hcl
# Policy: pave-checkout-api-production
path "secret/data/pave/checkout-api/production" {
  capabilities = ["read"]
}
```

The `vault-agent-injector` sidecar authenticates using Kubernetes service account tokens (Vault K8s auth method). Each pod's service account is bound to its service-specific Vault policy.

## Sidecar Injection

When a service declares secrets in `pave.yaml`:

```yaml
secrets:
  - vault_path: secret/data/pave/checkout-api/production
    mount: /etc/secrets
```

The Deploy Engine adds Vault annotations to the pod spec:

```yaml
annotations:
  vault.hashicorp.com/agent-inject: "true"
  vault.hashicorp.com/agent-inject-status: "update"
  vault.hashicorp.com/role: "pave-checkout-api-production"
  vault.hashicorp.com/agent-inject-secret-redis-password: "secret/data/pave/checkout-api/production"
  vault.hashicorp.com/agent-inject-template-redis-password: |
    {{- with secret "secret/data/pave/checkout-api/production" -}}
    {{ .Data.data.redis_password }}
    {{- end }}
```

**Result:** The `vault-agent` sidecar runs alongside the application container and writes secret values as files at the mount path:
```
/etc/secrets/redis-password     → contents: "actual-redis-password-value"
/etc/secrets/db-url             → contents: "postgres://..."
/etc/secrets/stripe-key         → contents: "sk_live_..."
```

The application reads secrets from files, not environment variables. This is a code change for services that currently use env vars — a migration guide is provided in the [onboarding docs](../experience/onboarding-flows.md#secrets-migration).

## Rotation Flow

```
pave secrets rotate redis-password --services checkout-api,cart-api,order-api
    │
    ▼
1. RBAC check: caller must be admin or owner of all affected services
    │
    ▼
2. Generate or accept new secret value
   - If --generate: Secrets Engine generates random value (32 bytes, base64)
   - If --value: use provided value (base64 encoded)
    │
    ▼
3. Write new version to Vault:
   vault kv put secret/pave/checkout-api/production redis_password="new-value"
   (Vault auto-increments version)
    │
    ▼
4. Record rotation in PostgreSQL:
   INSERT INTO secret_rotations (secret_path, secret_key, service_name,
       old_version, new_version, rotated_by)
   VALUES ('secret/data/pave/checkout-api/production', 'redis-password',
       'checkout-api', 5, 6, $user_id)
   -- one row per affected service
    │
    ▼
5. For each affected service:
   a. Rolling restart of vault-agent sidecar:
      kubectl rollout restart deployment/{service} --namespace={environment}
   b. Wait for rollout to complete (pods replaced one at a time)
   c. Poll sidecar health endpoint to confirm new version:
      GET http://{pod-ip}:8200/v1/sys/health → check vault-agent version metadata
    │
    ▼
6. After all services restarted:
   - If all sidecars confirm new version → set pickup_confirmed_at
   - If any sidecar still on old version after 5 min → alert:
     "⚠️ checkout-api still using redis-password v5 (expected v6)"
    │
    ▼
7. Write audit_log: secret.rotated
```

## Pickup Confirmation

After rotation, the Secrets Engine polls each affected pod to confirm it's running the new secret version:

```go
func ConfirmPickup(ctx context.Context, service string, env string, expectedVersion int) error {
    pods, err := k8sClient.CoreV1().Pods(env).List(ctx, metav1.ListOptions{
        LabelSelector: fmt.Sprintf("app=%s", service),
    })

    for _, pod := range pods.Items {
        // Read the vault-agent's cached secret version
        // via the sidecar's annotation or a custom health endpoint
        version, err := getSecretVersion(pod, secretPath)
        if err != nil {
            return fmt.Errorf("pod %s: failed to read secret version: %w", pod.Name, err)
        }
        if version < expectedVersion {
            return fmt.Errorf("pod %s: still on version %d (expected %d)", pod.Name, version, expectedVersion)
        }
    }
    return nil
}
```

**Polling schedule:**
- Check immediately after rollout completes
- If not confirmed: retry every 30 seconds for 5 minutes
- After 5 minutes: fire alert, mark rotation as `pickup_failed` in the alert

## Emergency Rotation Procedure

For critical secret compromises (e.g., leaked API key):

```bash
# Emergency: rotate immediately, don't wait for confirmation
pave secrets rotate stripe-key --services payments-api --emergency

# This:
# 1. Rotates the secret in Vault immediately
# 2. Force-restarts all pods (not rolling — kills all and recreates)
# 3. Alerts #platform-incidents channel
# 4. Skips the 5-min pickup confirmation window
```

**Emergency flag differences:**
- Normal rotation: rolling restart (zero-downtime, one pod at a time)
- Emergency rotation: force restart (brief downtime, all pods at once — speed > availability)

## Service Configuration for Secrets

```yaml
# pave.yaml — secrets section
secrets:
  - vault_path: secret/data/pave/checkout-api/production
    mount: /etc/secrets
    keys:
      - redis-password
      - db-url
      - stripe-key
    refresh_interval: 5m    # how often vault-agent checks for new versions
```

**`refresh_interval`:** Controls how often the vault-agent sidecar checks Vault for updated secret values. Lower values mean faster pickup but more Vault load. Default: 5 minutes. For secrets that rotate frequently (e.g., dynamic database credentials), set to 1 minute.

## Non-K8s Stacks — Known Gap

Docker Compose services (Gridline) cannot use vault-agent-injector because there's no sidecar injection mechanism in Docker Compose. Current workaround:

1. Secrets are stored as environment variables in a `.env` file on the host
2. Rotation requires: update `.env` → `docker-compose restart`
3. This is manual and not tracked by the Secrets Engine

**Impact:** Gridline's 3 services have no automated secret rotation and no pickup tracking. If a shared secret (e.g., Redis password) is rotated for K8s services, Gridline services must be manually updated separately.

**Mitigation plan:** When Gridline migrates to K8s (planned but no date), they'll get full secrets management. Until then, rotation runbooks include a manual step for Gridline.

## Security Considerations

| Concern | Mitigation |
|---------|-----------|
| Vault token compromise | Vault tokens are K8s service account tokens — short-lived, auto-rotated |
| Secret values in logs | Vault-agent writes secrets as files, not env vars. Files are 0600 permissions. Never logged. |
| Secret values in memory | Application reads from file on each use — not cached in memory (recommended pattern, not enforced) |
| Vault outage | Sidecar caches last-known secret. Application continues with cached value. Alert fires if Vault is unreachable for > 5 min. |
| Unauthorized rotation | RBAC check: only admin or service owner can rotate. All rotations audited. |
