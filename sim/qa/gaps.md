## NEW Gaps (Rounds 2-10)

### G-10: Client-side PHI purge has no automated proof

**Severity: High**

**Source:** S-04 (BOX-12, BOX-13)

BOX-12 and BOX-13 are "Proven (API) / Weak (client)". The API proof is solid — tokens are invalidated, requests return 410. But the core claim is about client-side behavior: DOM is purged, JavaScript memory is nullified, browser history is replaced, autocomplete data is cleared.

**The gap:** No automated browser test exists to verify the SessionPurge module. This is the most critical client-side proof in the entire system — the S-04 data breach happened because of inadequate client-side clearing. The fix (SessionPurge) has no automated verification that it works.

**Recommendation:**
1. Implement Playwright/Puppeteer E2E test: create session, load S3P, finalize, assert DOM has zero patient data elements, assert browser history shows S0, assert localStorage/sessionStorage empty.
2. This should be a CI gate — client deployment should not proceed if the purge test fails.
3. Visual regression test: screenshot after purge should match S0 baseline exactly.

### G-11: Medication per-item audit trail is absent

**Severity: Medium**

**Source:** S-06, acknowledged in Engineer constraints.md

The audit captures the entire medications list as old_value/new_value JSONB. Individual medication additions/removals are embedded in the list diff, not first-class audit events.

**The gap:** If a regulatory query asks "when was Lisinopril removed from this patient's medication list?", the answer requires parsing JSONB diffs across all audit entries. There is no indexed, queryable per-medication audit.

**Impact:** BOX-24 is proven for "medication list confirmed" events. But the granularity of audit may not satisfy a HIPAA/regulatory audit that asks about specific drug changes.

**Recommendation:** Accept for now as known debt. If regulatory requirement emerges (per Engineer constraints.md), a dedicated medications table with row-level tracking is needed. QA should flag this as a compliance risk to PM.

### G-12: Performance boxes are unproven without load test infrastructure

**Severity: High**

**Source:** S-09 (BOX-32, BOX-33, BOX-39)

BOX-32 (30 concurrent check-ins), BOX-33 (60 concurrent across locations), and BOX-39 (import doesn't degrade live system) all require load testing. No load test infrastructure exists yet.

**The gap:** These boxes are claimed by Engineering's architecture (horizontal scaling, read replicas, connection pooling, throttling) but have never been proven under actual load. The architecture is plausible but unverified.

**Recommendation:**
1. DevOps must provision a load test environment that mirrors production (same DB sizing, same connection pool, same number of service instances).
2. QA must run Runs 13 and 14 (load tests) before any scale event: S-05 (second location goes live), S-09 (Monday morning crush), S-10 (Riverside import).
3. Load test results become part of the proof registry — latency percentiles, error rates, connection pool utilization.

### G-13: Mobile pre-check-in link security at scale

**Severity: Medium**

**Source:** S-03 (BOX-08, BOX-11)

Pre-check-in links contain a 64-char token in the URL. The identity verification gate (DOB check with 3-attempt lockout) is the only defense for mobile.

**The gap:** At scale (S-05 + S-10: multiple locations, 4000+ patients), the attack surface grows. A motivated attacker could:
- Enumerate link tokens (64-char token makes this infeasible, but worth confirming entropy)
- Intercept SMS/email links in transit
- Social-engineer DOB from public records to bypass verification

The rate limit (5 req/min/IP) helps but is bypassable with distributed IPs.

**Recommendation:**
1. Confirm token entropy: 64 chars from what alphabet? Must be cryptographically random.
2. Consider adding a second verification factor for high-sensitivity data (medications, allergies).
3. Log and alert on: multiple locked links for different patients from similar IPs (coordinated attack signal).

### G-14: Dedup algorithm false negatives on name changes

**Severity: Medium**

**Source:** S-10, acknowledged in Engineer constraints.md

The dedup algorithm does not handle name changes (marriage, legal name change). A patient who married and changed their last name since their Riverside records were created will appear as two different patients.

**The gap:** No proof mechanism exists for this edge case. The algorithm is conservative (flags for human review at any match), but if the name is completely different and no other signals match (different phone, different address), the duplicate will be missed entirely.

**Impact:** BOX-37 (duplicates detected) is proven for its stated algorithm. But the algorithm has known blind spots.

**Recommendation:** Accept as known limitation. Add to S-10 import documentation: "Name changes since Riverside records may result in duplicate records. Recommend proactive outreach to Riverside patients to confirm identity after import." QA should track duplicate creation rate post-import as a degradation signal (new signal S-14).

### G-15: Concurrent finalization edge case — DB pool exhaustion

**Severity: Medium**

**Source:** S-07, DevOps negotiation, Engineer constraints.md

Engineer constraints.md states: "if connection pool is exhausted... concurrent prevention check fails (query times out) -> potential for duplicate sessions." BOX-O3 is the safety net (background check every minute).

**The gap:** The 1-minute gap between BOX-O3 checks means a duplicate session could exist for up to 60 seconds before detection. During that window, both sessions could reach finalization, and the optimistic locking would handle conflict. But if both sessions are at different locations with different receptionists, neither knows about the other until finalization fails.

**Impact:** The system handles this correctly (409 conflict, S8 resolution, S9 recovery). The gap is not in data integrity but in UX — the receptionist discovers the conflict only at finalization time, not at session creation time.

**Recommendation:** Reduce BOX-O3 check interval from 1 minute to 15 seconds during peak hours. Or: add a pre-finalization check (GET /checkins/{id} before POST finalize) that surfaces conflicts earlier.

### G-16: Import record error recovery is manual

**Severity: Low**

**Source:** S-10, Engineer constraints.md ("known debt")

If a record fails to import, it's flagged with error status but there is no automatic retry. Recovery requires manual re-import via paper entry (S13).

**The gap:** At 3200 records with a realistic error rate of 1-2%, that's 32-64 records requiring manual attention. No automated retry mechanism. No bulk error re-processing.

**Recommendation:** Accept for initial migration. Track error categories post-import. If most errors are transient (timeout, connection), add a retry mechanism. If most are data quality (validation failure), manual handling is correct.
